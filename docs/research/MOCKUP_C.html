Direction C — Contextual Drawer
Most discreet: safety lives inside the existing Filters modal. Cards carry an ambient color rail. Information is offered, never imposed.
Clean 5/5 Effective 3/5 Intuitive 3/5 Pleasing 5/5 Total 16/20
Why this direction exists
Safety is not the only thing Airbnb users care about — many are looking at price, beds, dates, instant book. The brief's §3 problems are real, but adding more first-class chrome to a page that's already dense risks degrading every other task. C respects the existing information hierarchy: cards gain a quiet color rail (peripheral signal), the Filters modal absorbs the safety tier control (where users already go to refine searches), and the inline drawer remains the single moment of "tell me more". Lowest visual cost. Highest sophistication. Best fit for users who want safety as one signal among many — not the dominant one.

Edge Cases
Edge · Modal selection "Very Safe only" → 0 results
Listings panel renders empty-state card: "No Very Safe stays match your filters" · body "Try adding Safe to see {N} more options." · CTA "Add Safe (B)" — opens modal pre-scrolled to safety section with "Safe" tier highlighted. Filters button dot pulses once for 300ms.
Edge · Drawer open + filter applied via modal
Modal save triggers drawer auto-collapse with 150ms fade. If listing remains in filtered set: rail returns to 4px default. If filtered out: scroll resets to top of panel (smooth).
Edge · Heatmap on + no safety filter applied
All four heatmap colors render at 0.25 alpha. Map legend visible. Cards continue to show their grade rails — direct visual mapping between rail color and map zone color.

Backward-from-Failure — P0 Components
Filters modal section: If users don't think to open Filters, they will fail Task 2 (filtering). Mitigation — the rails on cards create a chromatic problem (red/orange visible) which prompts the user to seek a control; the dot on the Filters button confirms it's the right place to look once safety has been engaged once.
Color rail: If users don't connect rail color to safety, they will fail Task 1. Mitigation — inline "Safety: A" text in metadata row provides explicit semantic label; rail + label + grade-color triple-encode the same signal.
Inline drawer: If users don't realize the meta text or rail is tappable, they will fail Task 3. Mitigation — cursor pointer + hover underline on meta text + tooltip "View safety details" on first hover per session.
Distinctiveness Log — what makes C unique
Safety filter has zero new chrome on the page itself — lives entirely inside the existing Filters modal. A adds a pill, B adds a bar, C adds nothing.
Card signal is a color rail (left border), not an overlay or chip — visible at the periphery, doesn't compete with photo or title.
Filters modal supports multi-select tiers (e.g. A + B). A and B are single-select.
Active-filter cue is a small dot on the Filters button — Airbnb's existing pattern for "filters engaged" is the same. Most native of the three.
Best fit when safety is one of many concerns — does not bias the user toward over-weighting it.
Designer Notes for Prototyper · Direction C
Internal Design Debate Summary
Q3 Filter Placement Three candidates: (a) sub-nav pill, (b) dedicated bar, (c) Filters modal. (c) wins for Direction C because the Filters modal already contains all the secondary refinement controls (price, beds, instant book) — placing safety there matches the user's existing mental model: "if I want to refine, I open Filters."
Q5 Badge Style Three candidates: (a) photo overlay, (b) metadata chip, (c) card border. (c) wins because C's whole stance is "ambient, not imposing" — a 4px color rail signals at the periphery without ever competing with the listing's primary content. A reviewer who isn't looking for safety can ignore the cards entirely; a reviewer who IS looking for safety can scan rail colors at high speed.
Trade-off Acknowledged Lowest discoverability of the three (Effective + Intuitive both 3/5). A first-time user who didn't know the feature exists may not find it — the only signal on the page (color rails) is non-obvious unless the user already knows safety grades exist. Mitigation: Filters modal "NEW" pill + first-visit pulsing toggle + map legend.

Final Falsification Pass
Failure mode: A user who came to the page specifically because they care about safety doesn't find the filter. They scan the page, see no safety control, conclude the feature doesn't exist, and either bounce or accept the default listing set without any filtering. The card rails read as decorative, not informational.
Mitigation: (1) Map legend in bottom-left of map is always visible — explains the color system unprompted. (2) On first visit per session, trigger a one-time tooltip on the Filters button: "New: filter by neighborhood safety" — dismissible, never repeats. (3) Inline meta text "Safety: A" provides a semantic label so the rail color is never the only encoding.
Exact Build Instructions for Prototyper
Filters modal: add a new section with id safety-filters after the Bedrooms section. Use existing Airbnb modal section pattern — 1px #EBEBEB dividers above and below, section title in 16px/600, four checkbox rows.
Card rail: use border-left: 4px solid var(--safety-color) with box-sizing: border-box on each card. The card's overall outer width remains 307px — the rail eats 4px of inner photo space (negligible visually).
Card meta safety text: add a span <span class="card-safety">Safety: <strong>A</strong></span> as the LAST flex child in the existing metadata row. The grade letter takes the safety-color via inline style. Whole span is clickable.
Filters button dot: absolute-positioned span on the existing button element. top: -2px; right: -2px; width: 10px; height: 10px; background: #1A7A4A; border-radius: 50%; box-shadow: 0 0 0 2px #FFFFFF;. Render conditionally based on safetyFilterCount > 0.
First-visit signal: read sessionStorage flag safety_intro_seen. If absent, show the Filters tooltip + heatmap-toggle pulse for 1 visit, then set the flag to true.
Multi-select: maintain activeSafetyTiers: Set<'A' | 'B' | 'C' | 'D'>. Filtering logic: card visible if activeSafetyTiers.size === 0 || activeSafetyTiers.has(card.grade).
